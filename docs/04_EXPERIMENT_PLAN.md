# 实验验证计划

最后更新：2026-06-07

## 1. 实验目标

验证以下核心命题：

> 基于原子知识点、推理链、技能树和诊断反馈的学习系统，是否能让零基础学习者以更高效率掌握基础财务报表阅读能力。

## 2. 实验场景

学习主题：基础财务报表阅读。

目标学习者：非财经专业学生，优先选择计算机、理工科或其他未系统学习过会计/财务的同学。

学习目标：学习者能够看懂简化版资产负债表、利润表、现金流量表，并能完成基本公司经营健康判断。

## 3. 为什么不先做小学生数学实验

原因：

1. 当前身份是计算机专业研究生，不方便稳定接触小学生样本。
2. K12 涉及家长信任、未成年人保护、合规、长期追踪等复杂问题。
3. 小学数学体系过大，不适合早期快速验证。
4. 成人/大学生实验更容易招募、执行和迭代。

## 4. 为什么选择财务报表实验

1. 对计算机专业同学足够陌生。
2. 知识结构清晰，适合技能树和推理链。
3. 题目答案相对明确，便于评分。
4. 7—14 天可完成学习闭环。
5. 对创业、投资、职场理解有现实价值。

## 5. 实验版本

### 第 0 轮：v0.2 本地验收（已完成）

样本：项目负责人自己 + 1 个测试学生账号。

目标：确认系统流程不报错、数据能保存、报告能导出。

验收结论：通过。

已确认：

```text
系统可以正常启动
5 条推理链都能打开
每条链的 8 个节点都能正常展示
学生可以提交推理回答
回答能写入 node_learning_records
通过时 node_status 变为 mastered，未通过时变为 weak
报告页能读取学习记录，不报错
CSV 能正常导出
一个测试学生能完整完成：前测 → 推理式学习 → 后测 → 报告
本轮未发现页面 bug、内容跳步、评分误判
```

### 第一轮：v0.2.1 内部试跑（当前阶段）

样本：1—3 人。

目标：在功能已通过验收的基础上，发现真实学习体验问题、内容跳步问题、评分误判问题，并观察前后测提升。

重点观察：

```text
学生是否知道下一步做什么
是否理解技能树/推理链
是否能完成前测和后测
推理式页面是否真正促进理解
expected_reasoning_points 判定是否误判明显
数据是否完整保存
```

### 第二轮：小规模验证

样本：6—12 人。

目标：初步验证学习效果。

可采用单组前后测，或简单对照组。

### 第三轮：对照实验

样本：10—30 人。

目标：比较 SkillTree 与普通学习方式的差异。

分组：

```text
实验组：使用 SkillTree Finance 学习
对照组：使用普通资料 / 视频 / ChatGPT 自学
```

## 6. 实验流程

当前代码支持的推荐流程：

```text
1. 填写学生信息
2. 完成前测
3. 进入推理式学习页，选择一条推理链
4. 按节点阅读核心问题、情境案例、引导问题、规则总结
5. 提交掌握验证解释
6. 系统根据 expected_reasoning_points 给出通过/未通过反馈
7. 完成后测
8. 查看报告并导出 CSV
```

当前代码暂未完整支持、需要人工补充的实验环节：

```text
学习前背景问卷
主观反馈问卷
一周后延迟测试页面
正式对照组材料分发
实验知情说明与统一操作手册
```

因此第 0 轮和第一轮试跑不要追求完整论文级实验设计，应先验证系统闭环和学习体验。


第一轮内部试跑的最低数据要求是：

1. 每个学生至少有一条完整前测记录。
2. 至少完成一条推理链的若干节点学习记录。
3. 至少有一条后测记录。
4. 报告页能读取并导出该学生数据。

## 8. 核心指标

### 8.1 学习效果指标

```text
前测分数
后测分数
前后测提升
延迟测试保持率
迁移题正确率
```

### 8.2 学习效率指标

```text
总学习时长
单位时间得分提升
每个节点平均学习时长
达到掌握所需尝试次数
```

### 8.3 诊断质量指标

```text
最常见错误类型
系统推荐是否被学生认可
薄弱节点是否能通过回退学习改善
```

### 8.4 主观体验指标

```text
技能树是否帮助理解路径：1-5 分
推理式学习是否比直接讲解更清晰：1-5 分
错误诊断是否准确：1-5 分
是否愿意继续使用：1-5 分
是否愿意推荐给他人：1-5 分
开放反馈
```

## 9. 前测/后测设计原则

前测与后测不能完全相同，但应覆盖同一批能力。

题型建议：

```text
选择题：测试基本概念
计算题：测试指标公式
判断题：测试概念辨析
简答题：测试解释能力
案例题：测试迁移应用
```

应覆盖：

```text
资产、负债、所有者权益
资产 = 负债 + 所有者权益
收入、成本、费用、利润
利润不等于现金流
资产负债表、利润表、现金流量表
毛利率、净利率、资产负债率
应收账款风险
公司健康判断
```

## 10. 延迟测试设计

时间：学习结束后一周。

目标：验证“牢牢掌握”，而不是短期记忆。

要求：

1. 不允许看资料。
2. 使用新案例。
3. 重点测试迁移能力。
4. 记录保持率。

## 11. 对照组设计

实验组：使用 SkillTree Finance。

对照组：使用普通学习资料，例如文章、视频、PPT 或自己问 ChatGPT。

两组控制：

```text
学习总时间尽量一致
学习主题一致
前后测试题一致
测试环境一致
```

比较：

```text
后测分数
提升幅度
学习时长
延迟测试保持率
主观路径清晰度
```

## 12. 反馈问卷建议

题目：

```text
1. 你觉得技能树是否帮助你理解学习路径？1-5 分
2. 你觉得推理式问题是否帮助你真正理解？1-5 分
3. 你觉得错误诊断是否准确？1-5 分
4. 你觉得学习效率是否比普通资料更高？1-5 分
5. 哪个知识节点最难？
6. 哪个功能最有帮助？
7. 你是否愿意继续使用？
8. 如果这个系统做成 7 天训练营，你是否愿意付费？可接受价格是多少？
9. 你希望它扩展到哪些领域？
10. 开放建议。
```

## 13. 实验成功标准

第一阶段不追求严谨论文级显著性，先看信号。

初步成功信号：

```text
多数学生能完整走完流程
前后测有明显提升
学生认为技能树能帮助理解路径
学生认为推理式学习比直接读定义更清晰
学生能在延迟测试中保留主要能力
至少部分学生表示愿意继续使用或付费尝试
```

更强成功信号：

```text
实验组单位时间提升明显高于对照组
延迟测试保持率高于对照组
学生能说清楚自己曾经卡在哪些节点
推荐路径能明显降低后续错误
```

## 14. 实验后复盘问题

每轮实验后回答：

1. 学生最容易卡在哪些节点？
2. 哪些节点太粗，需要继续拆？
3. 哪些推理链不自然？
4. 哪些题目无法真正测出掌握？
5. 学生是否真的使用了推荐路径？
6. 系统是否只是测评，而没有促进学习？
7. 是否值得进入下一轮样本扩大？

## 15. 当前实验优先级

当前先做：

```text
1. 固化 v0.2 功能验收通过状态
2. 准备 v0.2.1 内部试用记录表
3. 找 1—3 名真实用户完整试跑
4. 记录前测、后测、学习时长、薄弱节点、主观反馈
5. 重点观察解释不清、内容跳步、评分误判和学生困惑点
6. 根据试跑结果决定 v0.3 功能
```

暂缓：

```text
正式对照实验
大规模招募
复杂统计图
付费训练营
大模型自动评分
云部署
```

进入 6—12 人小规模验证前，必须先满足：

1. v0.2 功能验收已通过并完成版本固化。
2. 至少 1 名真实用户完整走完前测 → 推理式学习 → 后测 → 报告。
3. 至少收集 5 条真实用户反馈。
4. 已形成一份内部试跑问题清单和修复记录。
5. 能初步判断后测是否相对前测有提升，以及未提升的可能原因。
