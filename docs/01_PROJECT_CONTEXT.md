# SkillTree Finance 项目总览与上下文

最后更新：2026-06-07

## 1. 项目一句话定义

SkillTree Finance 是一个用于验证“知识图谱 + 原子知识点 + 推理链 + 掌握式学习 + AI/规则诊断”是否能提升学习效率的教育创业 MVP。

当前实验场景选择为：让非财经专业学生从零学习基础财务报表阅读。

## 2. 项目长期愿景

长期目标不是做一个普通 AI 家教，也不是做一个单纯的财务课，而是探索一种新的高效率学习系统：

> 把一门结构化知识拆成可推理的原子知识节点，让学习者像点亮游戏技能树一样，沿着最小认知负荷路径逐步掌握知识，并通过诊断、反馈、复习和迁移验证形成真正掌握。

## 3. 当前阶段

当前处于：MVP 原型开发与早期验证阶段。

已经形成的方向：

1. 第一版不直接做 K12 小学数学，因为实验对象难找，监管和家长信任门槛高。
2. 选择“零基础财务报表学习”作为近身同学可参与的成人学习实验场景。
3. 使用 Codex 开发本地 Python + Streamlit 原型。
4. 初始系统包含注册、前测、技能树、节点学习、诊断、后测、报告和数据导出。
5. 最新重要纠偏：当前原型不能只停留在“测试系统”，需要升级为“推理式学习系统”。

## 4. 为什么选择财务报表作为第一个场景

选择原因：

1. 对计算机专业研究生和身边同学来说通常是非专业领域，适合作为零基础学习实验。
2. 知识结构清晰，天然适合拆成技能树。
3. 学习结果容易通过题目、案例分析、前后测量化。
4. 周期短，能在 7—14 天内完成小规模实验。
5. 有现实吸引力，和创业、投资、商业理解、就业判断有关。

## 5. 核心假设

### 假设 1：学习路径结构化能降低认知负担

如果学生能看到知识节点之间的依赖关系，就更容易知道自己在哪里、为什么卡住、下一步应该学什么。

### 假设 2：原子知识点比粗粒度章节更适合高效学习

粗节点如“资产负债表”“利润表”过大，不适合直接学习。应该拆成更细的认知单元，如“企业为什么需要资源”“资源为什么一定有来源”“未来收钱的权利为什么也是资产”。

### 假设 3：推理链比灌输式讲解更有效

如果底层节点设计得足够好，学生学完第一层知识后，可以自然推出第二层知识。例如：

企业需要资源 → 资源有来源 → 资源总量等于来源总量 → 资产 = 负债 + 所有者权益 → 资产负债表。

### 假设 4：诊断反馈比单纯给答案更有价值

系统不仅要告诉学生题目错了，还要判断错因：概念混淆、前置知识缺失、利润现金流混淆、资产负债混淆、比率公式错误等。

### 假设 5：学习效果必须可验证

不能只看“学生觉得懂了”。必须通过前测、后测、延迟测试、迁移题、学习时长、错因统计和主观反馈验证效果。

## 6. 当前产品原则

1. 不做普通 AI 问答学习助手。
2. 不以“AI 替代老师”为宣传核心。
3. 不追求先做完整平台，而先验证单领域学习机制。
4. 不先做 UI 美化、排行榜、移动端、支付、云部署。
5. 优先验证“技能树学习是否真的提高效率”。
6. 产品核心不是讲解能力，而是学习过程控制能力。
7. 每个知识节点必须说明：依赖什么、能推出什么、解决什么认知问题、容易产生什么误解、如何验证掌握。

## 7. 项目当前最大变化

最初 MVP 偏向：

> 前测 → 技能树 → 做题 → 后测 → 报告。

最新判断认为这不足够，因为它更像测评系统。

项目应升级为：

> 原子知识点 → 推理链学习 → 引导式问题 → 学生自我解释 → 规则总结 → 掌握验证 → 后续节点推理。

因此 v0.2 的优先级应从“管理员页面/UI 美化/复杂报告”转为：

1. 微知识点重构；
2. 推理链字段设计；
3. 推理式学习页面；
4. 学生解释记录；
5. expected_reasoning_points 判断。

## 8. 当前建议的下一步

1. 保留现有 v0.1 测评基础设施。
2. 重构 `knowledge_graph.yaml`，从 20 个粗节点变成约 40 个微知识节点。
3. 新增推理链结构，先实现 5 条核心财报推理链。
4. 新增 `pages/3_ReasoningLesson.py`，支持核心问题、情境案例、引导问题、规则总结、掌握验证。
5. 第一版不接入复杂大模型判断，先用关键词或 expected_reasoning_points 做简单判断。
6. 后续再引入 LLM 个性化解释与简答题推理评分。
