# SkillTree Finance 项目总览与上下文

最后更新：2026-06-12

## 1. 项目一句话定义

SkillTree Finance 是一个用于验证“知识图谱 + 原子知识点 + 推理链 + 掌握式学习 + AI/规则诊断”是否能提升学习效率的教育创业 MVP。

当前实验场景选择为：让非财经专业学生从零学习基础财务报表阅读。

## 2. 项目长期愿景

长期目标不是做一个普通 AI 家教，也不是做一个单纯的财务课，而是探索一种新的高效率学习系统：

> 把一门结构化知识拆成可推理的原子知识节点，让学习者像点亮游戏技能树一样，沿着最小认知负荷路径逐步掌握知识，并通过诊断、反馈、复习和迁移验证形成真正掌握。

## 3. 当前阶段

当前处于：**v0.2 功能验收已通过，进入 v0.2.1 内部试用与学习效果观察阶段**。

根据代码审查与 v0.2 验收报告，当前系统已经具备以下能力：

1. 第一版仍选择“零基础财务报表学习”作为近身同学可参与的成人学习实验场景。

4. v0.2 推理式学习已初步落地：`knowledge_graph.yaml` 已重构为 40 个微知识节点，形成 5 条核心财报推理链。
5. `pages/3_ReasoningLesson.py` 已存在，可以按推理链展示核心问题、情境案例、引导问题、规则总结、掌握验证题，并保存学生解释。
6. v0.2 功能验收已通过：系统启动、5 条推理链打开、40 个节点展示、推理回答提交、`node_learning_records` 写入、`node_status` 更新、报告页读取、CSV 导出、完整流程均已确认可用。
7. 当前最重要的问题已经从“功能是否能跑通”转为“学习效果是否真实成立”。

## 4. 为什么选择财务报表作为第一个场景

选择原因：

1. 对计算机专业研究生和身边同学来说通常是非专业领域，适合作为零基础学习实验。
2. 知识结构清晰，天然适合拆成技能树。
3. 学习结果容易通过题目、案例分析、前后测量化。
4. 周期短，能在 7—14 天内完成小规模实验。
5. 有现实吸引力，和创业、投资、商业理解、就业判断有关。

## 5. 核心假设

### 假设 1：学习路径结构化能降低认知负担

如果学生能看到知识节点之间的依赖关系，就更容易知道自己在哪里、为什么卡住、下一步应该学什么。

### 假设 2：原子知识点比粗粒度章节更适合高效学习

粗节点如“资产负债表”“利润表”过大，不适合直接学习。应该拆成更细的认知单元，如“企业为什么需要资源”“资源为什么一定有来源”“未来收钱的权利为什么也是资产”。

### 假设 3：推理链比灌输式讲解更有效

如果底层节点设计得足够好，学生学完第一层知识后，可以自然推出第二层知识。例如：

企业需要资源 → 资源有来源 → 资源总量等于来源总量 → 资产 = 负债 + 所有者权益 → 资产负债表。

### 假设 4：诊断反馈比单纯给答案更有价值

系统不仅要告诉学生题目错了，还要判断错因：概念混淆、前置知识缺失、利润现金流混淆、资产负债混淆、比率公式错误等。

### 假设 5：学习效果必须可验证

不能只看“学生觉得懂了”。必须通过前测、后测、延迟测试、迁移题、学习时长、错因统计和主观反馈验证效果。

## 6. 当前产品原则

1. 不做普通 AI 问答学习助手。
2. 不以“AI 替代老师”为宣传核心。
3. 不追求先做完整平台，而先验证单领域学习机制。
4. 不先做 UI 美化、排行榜、移动端、支付、云部署。
5. 优先验证“技能树学习是否真的提高效率”。
6. 产品核心不是讲解能力，而是学习过程控制能力。
7. 每个知识节点必须说明：依赖什么、能推出什么、解决什么认知问题、容易产生什么误解、如何验证掌握。

## 7. 项目当前最大变化

最初 MVP 偏向：

> 前测 → 技能树 → 做题 → 后测 → 报告。

这个形态更像测评系统，不足以证明“推理式学习系统”的价值。

当前代码已经开始转向：

> 原子知识点 → 推理链学习 → 引导式问题 → 学生自我解释 → 规则总结 → 掌握验证 → 后续节点推理。

本次代码审查与验收确认，v0.2 的关键结构已经初步实现并通过功能验收：

1. 微知识点重构：`data/knowledge_graph.yaml` 中已有 40 个节点；
2. 推理链字段：每个节点包含 `chain`、`layer`、`prerequisites`、`derives`、`contrasts` 等字段；
3. 推理式学习页面：`pages/3_ReasoningLesson.py` 已实现；
4. 学生解释记录：通过 `node_learning_records` 表保存；
5. expected_reasoning_points 判断：当前采用关键词/要点匹配，达到 60% 命中即判定通过。

因此，项目的当前最大变化不再是“要不要做推理式学习”或“功能能否跑通”，而是：

> 已经有了通过功能验收的推理式学习原型，下一步要通过真实用户试用证明它是否有学习效果。

## 8. 当前建议的下一步

当前不建议继续盲目加新功能。由于 v0.2 功能验收已经通过，下一步应进入 **v0.2.1 内部试用与学习效果观察**。

优先顺序如下：

1. **固化 v0.2 版本**：提交当前代码与文档，建议打 `v0.2` 或 `v0.2-accepted` 标签。
2. **更新 README 与 docs 状态**：把“v0.2 待验收”统一改为“v0.2 功能验收通过，进入 v0.2.1 内部试用”。
3. **准备 1—3 人内部试用**：让真实同学完整完成前测 → 推理式学习 → 后测 → 报告。
4. **记录学习效果数据**：前测分数、后测分数、学习时长、薄弱节点、反复答错节点、主观反馈。
5. **重点观察内容问题**：解释是否清楚、推理链是否跳步、expected_reasoning_points 是否误判。
6. **试用后再决定 v0.3**：根据真实卡点决定是否做反馈问卷、数据完整性检查、管理员页、分析脚本或 LLM 辅助评分。
7. **可并行设计 LLM Shadow Evaluator**：只做影子评分，不改变学生状态更新，用于比较规则评分与语义评分的差异。

一句话：

> v0.2 已经证明“系统能跑通”，v0.2.1 要证明“这个系统是否真的帮助人学会”。

## 9. 大模型 API 在当前阶段的位置

根据 v0.2 验收结果，当前系统没有必须依赖 DeepSeek、OpenAI、通义、智谱等外部大模型 API 才能完成的任务。系统主流程已经可以在本地规则化条件下跑通。

因此，当前阶段对大模型 API 的定位是：

> 大模型 API 是增强层，不是基础依赖；用于增强语义评分、个性化反馈、学习报告总结、错因分类、题目变式生成和知识图谱审查，而不替代技能树、推理链和掌握验证机制。

短期最值得尝试的是 **LLM Shadow Evaluator**：

```text
规则评分继续决定 mastered / weak
大模型 API 只做影子评分和反馈建议
记录 rule_result 与 llm_result 的差异
人工复核冲突样本
再决定是否让 LLM 进入正式评分链路
```

当前不建议直接开放自由聊天式 AI 家教，也不建议让大模型自动生成课程后直接发布。原因是这会干扰当前最重要的实验变量：技能树与推理链本身是否有效。


## 10. 模拟学生系统与课程自博弈优化的新方向

本轮新增重要想法：可以用多个大模型模拟不同类型学生，让它们在受控条件下长期试学当前知识图谱和推理链，并把失败样本反馈给系统，用于发现知识点拆分、前置关系、推理链、引导问题、掌握验证题和评分规则中的缺陷。

该方向的准确定位不是“让大模型真的不依赖自身知识学会财报”，因为通用大模型本身可能已经具备财务知识，很难严格证明它没有使用预训练知识。更合理的定位是：

> 用多模型模拟学生，对知识图谱和推理链进行自动压力测试，找出课程结构和掌握验证中的漏洞。

推荐名称：

```text
Synthetic Student Lab
模拟学生学习实验室
课程自博弈优化模块
```

它的价值在于：在真实学生大规模使用之前，先低成本发现明显的课程设计问题，例如：

```text
节点过粗
前置知识缺失
推理链跳步
引导问题无效
掌握验证题太浅
评分规则被关键词欺骗
补救路径推荐无效
迁移题不足
```

当前阶段的原则：

1. 不把该模块并入 v0.2.1 主流程，避免干扰真实学生试用。
2. v0.2.1 仍以 1—3 名真实用户试跑为最高优先级。
3. 模拟学生系统可作为 v0.3 起的离线课程质检工具，而不是面向学生的核心学习功能。
4. 后台长期运行可以探索，但必须是“受控自博弈 + 候选修改 + 回归测试 + 人工审核”，不能让大模型自动无限修改正式知识图谱。

推荐的长期闭环：

```text
当前知识图谱版本
→ 多类模拟学生试学
→ 生成回答与学习轨迹
→ 独立评估器判断掌握情况
→ 失败归因
→ 生成课程修改建议
→ 回归测试
→ 形成候选版本
→ 人工审核
→ 合并到正式知识图谱
```

与大模型 API 的关系：该模块需要 LLM API 支持，但应优先用于离线开发者工具。它和 LLM Shadow Evaluator 的区别是：

```text
LLM Shadow Evaluator：评估真实学生回答。
Synthetic Student Lab：模拟学生学习过程，压力测试课程结构。
```

因此，项目当前新形成的中期产品判断是：

> SkillTree 不仅要能教学生，还要能模拟学生学习过程，持续发现课程结构问题，并辅助教师迭代知识图谱与推理链。
