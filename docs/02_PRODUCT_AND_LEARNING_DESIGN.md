# 产品与学习机制设计文档

最后更新：2026-06-07

## 1. 产品定位

SkillTree Finance 不是普通财务课程，也不是普通 AI 家教，而是一个“推理式技能树学习系统”。

核心目标：

> 让学生从底层概念出发，通过一条可推理、可诊断、可验证的学习路径，逐步掌握财务报表阅读能力。

## 2. 产品核心机制

系统应由以下机制构成：

1. 原子知识点：把知识拆到足够小，每个节点只解决一个认知问题。
2. 推理链：节点之间不仅是章节关系，而是“可由前置节点推出后续节点”的逻辑关系。
3. 技能树：展示学生当前已掌握、薄弱、未学习、需要复习的节点。
4. 诊断系统：根据学生错误定位薄弱前置节点和错误类型。
5. 掌握验证：通过标准题、变式题、解释题、迁移题判断是否真正掌握。
6. 学生模型：记录学生在哪些节点真会、假会、快忘、混淆、迁移困难。
7. 路径推荐：基于当前状态推荐下一步最值得学习的节点。

## 3. 从“知识树”升级为“推理图”

早期知识树容易变成章节目录。例如：

```text
资产
负债
所有者权益
资产负债表结构
利润表结构
现金流量表结构
```

这种拆法过粗，不能保证学生真正理解。

应升级为推理依赖图：

```text
企业需要资源
→ 资源有来源
→ 来源分为借来的和自己投入的
→ 企业控制的资源叫资产
→ 借来的资源来源叫负债
→ 股东投入或留存的来源叫所有者权益
→ 资源总量一定等于来源总量
→ 资产 = 负债 + 所有者权益
→ 资产负债表
```

## 4. 知识节点设计原则

每个节点都应满足以下要求：

1. 足够原子化：只解决一个小问题。
2. 有核心问题：用问题驱动学习，而不是先给定义。
3. 有现实情境：用生活/商业案例降低认知负担。
4. 有引导问题：让学生自己推出规则。
5. 有规则总结：把直觉上升为正式概念。
6. 有常见误解：提前暴露容易混淆点。
7. 有掌握验证：要求学生证明自己理解。
8. 有推理出口：说明掌握该节点后能推出什么。

## 5. 推荐节点字段结构

```yaml
id: asset_001
title: 为什么现金是资产
type: atomic_concept
layer: 1
chain: balance_sheet_reasoning
prerequisites:
  - resource_001
derives:
  - asset_002
contrasts:
  - cashflow_001
core_question: 企业手里的现金为什么算一种资源？
scenario: 小明开奶茶店，手里有 10 万元现金，可以用来买设备、买原材料和支付工资。
guiding_questions:
  - 小明能控制这 10 万元吗？
  - 这 10 万元能帮助未来经营吗？
  - 如果没有现金，奶茶店还能顺利开业吗？
rule_summary: 能被企业控制，并能带来未来经济利益的资源，可以被看作资产。
common_misconceptions:
  - 只有固定资产才是资产
  - 资产就是现金
mastery_question:
  question: 为什么奶茶店账户里的 10 万元可以被看作资产？
expected_reasoning_points:
  - 企业可以控制这笔现金
  - 现金能用于未来经营
  - 现金能带来未来经济利益
```

## 6. 节点类型

建议支持四类节点：

### 6.1 原子概念节点 atomic_concept

用于建立最小概念。

例：
- 企业为什么需要资源
- 什么是未来经济利益
- 为什么现金是资产

### 6.2 推理规则节点 reasoning_rule

用于从多个概念推出关系。

例：
- 资源总量等于来源总量
- 资产 = 负债 + 所有者权益
- 利润 = 收入 - 成本 - 费用

### 6.3 对比辨析节点 contrast

用于处理容易混淆的概念。

例：
- 利润不等于现金流
- 资产不等于现金
- 收入不等于收款

### 6.4 应用迁移节点 application

用于综合判断。

例：
- 盈利但没现金是否危险
- 应收账款过高说明什么
- 如何判断公司是否健康

## 7. 知识图谱边类型

原来只有 prerequisite 不够，建议增加：

1. `prerequisite`：前置知识。
2. `derives`：由当前节点可以推出的后续节点。
3. `contrasts`：容易混淆、需要对比的节点。
4. `applies_to`：当前概念可以应用到哪些问题。
5. `misconception_of`：当前节点用于纠正哪个误解。

## 8. 五条核心财报推理链

### A. 从资源到资产负债表

```text
A1 企业需要资源才能经营
A2 资源有不同来源
A3 来源可以分为借来的和自己投入的
A4 企业控制的资源叫资产
A5 借来的资源来源叫负债
A6 股东投入或留存的来源叫所有者权益
A7 资源总量一定等于来源总量
A8 资产 = 负债 + 所有者权益
A9 资产负债表就是记录某一时点资源及其来源的表
A10 资产负债率可以衡量资源中有多少来自借款
```

### B. 从交易到利润表

```text
B1 企业通过卖产品或服务获得收入
B2 获得收入通常需要付出成本
B3 收入减去直接成本得到毛利
B4 企业还需要支付销售、管理等费用
B5 收入扣除成本和费用后得到利润
B6 利润表记录一段时间内企业是否赚钱
B7 毛利率反映产品本身赚钱能力
B8 净利率反映整体经营赚钱能力
```

### C. 从收付款到现金流量表

```text
C1 企业赚钱不等于马上收到现金
C2 现金流只关心钱有没有实际进出
C3 经营活动现金流来自日常经营
C4 投资活动现金流来自买卖长期资产或投资
C5 筹资活动现金流来自借钱、还钱、股东投入等
C6 现金流量表记录一段时间内现金如何变化
C7 经营现金流比净利润更能反映现金质量
```

### D. 从赊销到应收账款风险

```text
D1 企业可以先卖货后收钱
D2 先卖货后收钱会形成应收账款
D3 应收账款是未来收钱的权利
D4 应收账款增加会让收入增加但现金不增加
D5 应收账款过高可能说明回款风险
D6 利润增长但应收账款大增，需要警惕利润质量
```

### E. 从综合数据到公司健康判断

```text
E1 公司健康不能只看利润
E2 要同时看盈利能力、偿债压力、现金质量
E3 盈利能力看毛利率和净利率
E4 偿债压力看资产负债率
E5 现金质量看经营现金流和应收账款
E6 三类指标互相印证，才能形成判断
```

## 9. 推理式学习页面设计

每个节点的学习流程不应是“解释 + 题目”，而应是：

1. 核心问题：提出认知冲突或关键问题。
2. 情境案例：给一个现实情境。
3. 引导问题：让学生逐步判断。
4. 学生自我解释：让学生写出自己的推理。
5. 系统反馈：判断是否覆盖 expected_reasoning_points。
6. 规则总结：将推理结果整理为正式概念。
7. 后续推理：展示“你现在可以推出什么”。
8. 掌握验证：通过题目或解释题验证。

## 10. 一个完整推理学习示例

目标：理解“利润增加不等于现金增加”。

推理路径：

```text
卖出商品不一定马上收到钱
→ 未来收钱的权利也是一种资源
→ 这种权利叫应收账款
→ 收入记录的是赚到，不是收到
→ 现金流记录的是实际现金进出
→ 所以利润增加不等于现金增加
```

教学流程：

1. 提问：为什么企业卖出商品后，没收到钱，也可能确认收入？
2. 情境：奶茶店给公司客户送了 1000 元饮品，对方下个月付款。
3. 引导：东西交付了吗？钱收到吗？未来有收钱权利吗？利润会变吗？现金会变吗？
4. 学生解释：赚到钱和收到钱的区别是什么？
5. 系统总结：利润表看收入与费用，现金流量表看现金进出。
6. 验证：给另一家公司赊销案例，判断利润和现金流变化。

## 11. 产品与普通 AI 学习助手的区别

普通 AI：

```text
你不会，我给你讲。
```

SkillTree：

```text
你不会，我判断你缺哪个前置节点；
我把知识拆到更小；
让你用已知知识推出新知识；
再验证你是否真的能迁移。
```

这就是本项目最核心的产品哲学。
