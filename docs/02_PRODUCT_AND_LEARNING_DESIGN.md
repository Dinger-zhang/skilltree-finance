# 产品与学习机制设计文档

最后更新：2026-06-12

## 1. 产品定位

SkillTree Finance 不是普通财务课程，也不是普通 AI 家教，而是一个“推理式技能树学习系统”。

核心目标：

> 让学生从底层概念出发，通过一条可推理、可诊断、可验证的学习路径，逐步掌握财务报表阅读能力。

## 2. 产品核心机制

系统应由以下机制构成：

1. 原子知识点：把知识拆到足够小，每个节点只解决一个认知问题。
2. 推理链：节点之间不仅是章节关系，而是“可由前置节点推出后续节点”的逻辑关系。
3. 技能树：展示学生当前已掌握、薄弱、未学习、需要复习的节点。
4. 诊断系统：根据学生错误定位薄弱前置节点和错误类型。
5. 掌握验证：通过标准题、变式题、解释题、迁移题判断是否真正掌握。
6. 学生模型：记录学生在哪些节点真会、假会、快忘、混淆、迁移困难。
7. 路径推荐：基于当前状态推荐下一步最值得学习的节点。

## 3. 从“知识树”升级为“推理图”

早期知识树容易变成章节目录。例如：

```text
资产
负债
所有者权益
资产负债表结构
利润表结构
现金流量表结构
```

这种拆法过粗，不能保证学生真正理解。

应升级为推理依赖图：

```text
企业需要资源
→ 资源有来源
→ 来源分为借来的和自己投入的
→ 企业控制的资源叫资产
→ 借来的资源来源叫负债
→ 股东投入或留存的来源叫所有者权益
→ 资源总量一定等于来源总量
→ 资产 = 负债 + 所有者权益
→ 资产负债表
```

## 4. 知识节点设计原则

每个节点都应满足以下要求：

1. 足够原子化：只解决一个小问题。
2. 有核心问题：用问题驱动学习，而不是先给定义。
3. 有现实情境：用生活/商业案例降低认知负担。
4. 有引导问题：让学生自己推出规则。
5. 有规则总结：把直觉上升为正式概念。
6. 有常见误解：提前暴露容易混淆点。
7. 有掌握验证：要求学生证明自己理解。
8. 有推理出口：说明掌握该节点后能推出什么。

## 5. 当前代码中的节点字段结构

本次代码审查确认，`data/knowledge_graph.yaml` 已经采用 v0.2 微知识点结构。`src/graph.py` 要求每个节点必须包含以下字段：

```yaml
id: resource_controlled_by_company
title: 企业控制的资源
layer: 1
chain: A. 从资源到资产负债表
type: concept
prerequisites: []
derives:
  - future_economic_benefit
contrasts: []
core_question: 企业为什么需要能被自己控制的资源？
scenario: 小明开奶茶店，需要现金、设备、原材料和店铺才能经营。
guiding_questions:
  - 哪些东西是小明能自己支配使用的？
  - 哪些东西能帮助奶茶店未来经营？
rule_summary: 能被企业控制并帮助未来经营的资源，是理解资产的起点。
common_misconceptions:
  - 只要值钱就是企业资源
mastery_question: 为什么企业不能把别人的设备直接算作自己的资源？
expected_reasoning_points:
  - 企业必须能控制资源
  - 资源要能服务未来经营
```

需要注意两点：

1. 当前 `mastery_question` 是字符串字段，不是嵌套对象。
2. `src/graph.py` 会自动补充旧页面兼容字段：`level`、`learning_objective`、`explanation`、`exercises`、`mastery_questions`，使旧的 `3_LearnNode.py` 仍能读取新图谱。

## 6. 当前代码中的节点类型

当前 `type` 字段是普通字符串，代码不强制枚举，但 `knowledge_graph.yaml` 中实际使用了以下类型：

```text
concept
inference
rule
classification
contrast
calculation
analysis
risk_signal
link
process
synthesis
```

这些类型可以按学习功能理解：

1. `concept`：建立最小概念，例如“三张表的分工”。
2. `inference` / `rule`：从前置概念推出规则，例如“会计恒等式”。
3. `classification`：训练分类判断，例如“现金流入/经营活动/投资筹资活动”。
4. `contrast`：处理混淆点，例如“收入不等于收款”“权责发生制与现金制”。
5. `calculation`：训练基础计算，例如毛利、净利润、营运资本。
6. `analysis` / `risk_signal` / `synthesis`：训练综合分析和风险判断。
7. `link` / `process`：训练三张表勾稽或分析步骤。

后续是否需要统一成更少的枚举类型，应在 1—3 人试跑后决定。当前不建议为了“类型命名整齐”而打断实验验证。

## 7. 当前代码支持的知识图谱边类型

当前已经实现并被 `src/graph.py` 校验的边字段有三类：

1. `prerequisites`：前置节点。用于说明学习当前节点前应先理解什么。
2. `derives`：可推出的后续节点。用于通过后提示“你可以推出的下一个节点”。
3. `contrasts`：容易混淆、需要对比的节点。当前数据已有字段，但页面尚未充分利用。

当前暂未实现为代码功能的扩展边包括：

```text
applies_to
misconception_of
```

因此短期不要在数据中新增大量未被页面使用的边类型。v0.2 阶段应先验证 `prerequisites + derives + contrasts` 是否足够支撑学习路径。

## 9. 当前推理式学习页面实现

当前 `pages/3_ReasoningLesson.py` 已实现 v0.2 最小推理学习页，流程如下：

1. 学生在侧边栏登记姓名、学号、班级。
2. 选择一条推理链。
3. 页面按链内节点顺序展示学习内容。
4. 每个节点展示：核心问题、情境案例、引导问题、规则总结、常见误区、掌握验证题。
5. 学生提交“你的推理回答”。
6. 系统用 `expected_reasoning_points` 做关键词/要点匹配。
7. 当前通过规则：命中至少 60% 的预期推理点。
8. 通过后将节点状态设为“已掌握”，未通过则设为“薄弱”。
9. 系统把学生回答、正确要点、是否通过、用时、推荐节点写入 `node_learning_records`。
10. 通过后展示 `derives` 指向的后续节点。

当前实现的主要限制：

1. 评分是关键词/子串匹配，容易出现“关键词命中但不理解”或“同义表达未命中”。
2. `contrasts` 字段尚未被充分用于页面反馈。
3. 推理链可以跳转浏览，尚未强制按前置节点解锁。
4. 还没有教师端人工复核学生解释。
5. 还没有独立反馈问卷与延迟测试页面。

由于 v0.2 功能验收已经通过，下一步不是马上接入大模型，而是通过 1—3 名真实学生试跑观察这个最小机制是否真的有效。

## 10. 一个完整推理学习示例

目标：理解“利润增加不等于现金增加”。

推理路径：

```text
卖出商品不一定马上收到钱
→ 未来收钱的权利也是一种资源
→ 这种权利叫应收账款
→ 收入记录的是赚到，不是收到
→ 现金流记录的是实际现金进出
→ 所以利润增加不等于现金增加
```

教学流程：

1. 提问：为什么企业卖出商品后，没收到钱，也可能确认收入？
2. 情境：奶茶店给公司客户送了 1000 元饮品，对方下个月付款。
3. 引导：东西交付了吗？钱收到吗？未来有收钱权利吗？利润会变吗？现金会变吗？
4. 学生解释：赚到钱和收到钱的区别是什么？
5. 系统总结：利润表看收入与费用，现金流量表看现金进出。
6. 验证：给另一家公司赊销案例，判断利润和现金流变化。

## 11. 产品与普通 AI 学习助手的区别

普通 AI：

```text
你不会，我给你讲。
```

SkillTree：

```text
你不会，我判断你缺哪个前置节点；
我把知识拆到更小；
让你用已知知识推出新知识；
再验证你是否真的能迁移。
```

这就是本项目最核心的产品哲学。



## 12. 模拟学生学习实验室设计

模拟学生学习实验室不是面向学生的学习页，而是面向课程设计者和开发者的内容质检模块。

核心目标：

> 用多个大模型模拟不同学习者，长期试学知识图谱和推理链，发现真实学生可能遇到的课程结构问题。

### 12.1 模拟学生画像

第一版可以设计 5—8 类学生画像：

```text
零基础学生：只能依赖课程材料推理。
死记硬背型学生：能复述规则，但不会迁移。
误解型学生：容易混淆收入/收款、利润/现金流、资产/现金。
跳步型学生：喜欢直接给结论，中间推理不完整。
聪明但急躁型学生：能猜对答案，但解释缺关键步骤。
迁移困难型学生：原例题会做，换案例就失败。
反例挑战型学生：专门寻找规则边界和例外情况。
关键词堆砌型学生：堆出关键词但逻辑不成立。
```

### 12.2 可以发现的问题类型

模拟学生系统重点发现：

```text
节点是否过粗
是否缺少 prerequisite
是否存在 derives 跳步
核心问题是否不能引出规则
引导问题是否无效
expected_reasoning_points 是否粒度不一致
mastery_question 是否只测复述
迁移题是否不足
评分规则是否被关键词欺骗
推荐补救节点是否无效
```

### 12.3 推荐闭环

```text
knowledge_graph.yaml
→ synthetic student runner
→ 学生画像按节点学习并回答
→ judge 评估推理完整度
→ failure analyzer 归因
→ graph patch generator 生成候选修改
→ regression eval 回归测试
→ 人工审核后合并
```

### 12.4 不能自动无限修改正式课程

模拟学生可以长期后台运行，但只能生成候选建议，不能直接自动上线。必须设置：

```text
版本控制
修改 diff
回归测试
隐藏迁移题测试
事实准确性检查
成本上限
人工审核
真实学生数据校准
```

主要风险：

```text
过拟合模拟学生
过拟合 LLM judge
课程内容被大模型错误建议污染
API 成本失控
多个模型互相自嗨但真实学生无改善
```

因此，该模块的产品定位应是：

> 课程自动质检和改进建议系统，而不是自动自治课程生成系统。


## 13. 大模型 API 在产品机制中的作用

大模型 API 不应替代 SkillTree 的核心产品机制。系统的核心仍是：

```text
知识图谱结构
原子知识点
推理链
掌握验证
学生状态模型
学习路径控制
```

大模型 API 更适合作为语义增强器，重点增强以下环节：

1. **解释题语义评分**：判断学生回答是否真正覆盖 expected_reasoning_points，而不是只做关键词匹配。
2. **个性化反馈**：学生未通过时指出缺失推理点、常见误解和下一步思考方向。
3. **错因分类**：识别概念缺失、概念混淆、只背定义没有推理、收入与收款混淆、利润与现金流混淆等错误类型。
4. **学习报告总结**：把前测、后测、节点状态、薄弱节点和学习记录转成自然语言诊断报告。
5. **变式题和迁移题生成**：为每个节点生成基础题、反例题、案例题和迁移题，但必须人工审核后进入题库。
6. **节点内容润色**：检查 scenario、guiding_questions、rule_summary、mastery_question 是否清楚。
7. **知识图谱审查**：发现节点跳步、重复、过粗、缺少前置关系或推理关系不自然的问题。

### 13.1 不建议过早开放自由聊天

普通自由聊天式 AI 家教会带来三个问题：

```text
学生可能绕过技能树直接问答案
学习效果来源会被大模型讲解能力污染
系统难以判断学生到底掌握了哪个知识节点
```

如果未来要做 AI 助教，应采用受控节点内助教：

```text
只围绕当前 node_id 回答
不能直接给标准答案
优先使用苏格拉底式引导
反馈必须依据当前节点的 rule_summary 和 expected_reasoning_points
```

### 13.2 当前推荐策略

v0.2.1 阶段推荐策略：

```text
主流程：规则系统继续运行
状态更新：仍由规则评分决定
大模型：只做影子评分、反馈建议和开发者分析
数据：记录 rule_result 与 llm_result 的差异
决策：人工复核后再决定是否进入 v0.3 正式功能
```

## 14. Synthetic Student Lab 当前课程质检发现（2026-06-12）

B 链真实模拟实验显示，当前课程最值得优先小修的节点是：

```text
accrual_vs_cash：权责发生制 / 现金制边界不够稳；
net_profit：净利润形成机制与现金差异机制需要更显式；
gross_margin：毛利、净利润、现金之间的边界需要强化。
```

评分层面发现：旧 `expected_reasoning_points` 字面匹配过严，容易漏判同义表达；enhanced scorer v2 已加入语义归一和矛盾答案惩罚，但后续仍需人工抽查防止 false pass。

设计原则更新：课程 patch 应只修补明确由实验暴露出的认知缺口，不应为了模拟学生指标而过拟合课程内容。
